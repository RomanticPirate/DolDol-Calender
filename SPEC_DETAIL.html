레이아웃 어긋남 방지 월을 넘길 때 달력 높이가 변하지 않도록 다음 규칙을 적용합니다.
규칙설명
고정 행 수항상 6행(42셀) 표시. 5행만 필요한 월에도 다음 달 패딩으로 6행 채움
셀 크기aspect-ratio 사용하지 않음. 고정 min-height로 세로 높이 확보 (일정 미리보기 공간)
그리드 갭일정한 gap 유지. 월 변경 시에도 동일

4.1 라벨 렌더링 규칙

항목규칙

동기화 규칙 (Firebase 프로덕션)

12.1 동기화 원칙

최우선 원칙: 데이터 유실 절대 금지 어떤 상황에서도 사용자 데이터가 소멸되어서는 안됩니다. 충돌 시 양쪽 데이터를 모두 보존합니다.
원칙설명
일정별 updatedAt 비교각 일정의 updatedAt 타임스탬프로 어느 쪽이 최신인지 판별
최신 우선같은 일정 ID면 updatedAt이 더 최근인 쪽이 승리
삭제 기록 관리삭제된 일정은 deleteLog에 기록. 동기화 시 삭제 기록도 비교
충돌 보존양쪽 모두 수정된 경우 → 두 버전을 모두 보존하고 사용자에게 선택 UI 제공
쓰기 전 백업서버에 쓰기 전 로컬 데이터를 ourday_backup에 백업

12.2 동기화 프로세스 (ToDo 앱 규칙 기반)

1. 로컬 백업 생성→ 2. 서버 데이터 fetch→ 3. 일정별 updatedAt 비교→ 4. 병합 결과 생성→ 5. 병합 검증→ 6. 로컬 + 서버 동시 쓰기

12.3 병합 규칙 상세

상황처리
서버에만 있는 일정로컬에 추가 (단, deleteLog에 있으면 삭제된 것이므로 무시)
로컬에만 있는 일정서버에 추가
양쪽 모두 존재, 한쪽만 수정최신(updatedAt 큰 쪽) 채택
양쪽 모두 존재, 양쪽 모두 수정충돌! → 두 버전 보존, 사용자에게 diff UI로 선택 요청
한쪽 삭제, 한쪽 수정수정본 보존 (삭제보다 수정 우선)
양쪽 모두 삭제삭제 확정

12.4 병합 검증

병합 결과를 쓰기 전 다음을 검증합니다:

일정 수 감소 체크병합 후 일정 수가 병합 전(로컬 + 서버 합집합)보다 적으면 에러 → 쓰기 중단
빈 데이터 체크병합 결과가 빈 배열이면 쓰기 중단 (데이터 소멸 방지)
ID 중복 체크같은 ID의 일정이 2개 이상이면 에러

데이터 유실 방지 설계

13.1 방어 계층

계층보호 대상방법
L1. 쓰기 전 백업localStorage 손상save() 호출 전 현재 데이터를 ourday_backup에 복사
L2. 쓰기 후 검증쓰기 실패save() 후 즉시 read-back하여 데이터 무결성 확인. 실패 시 backup에서 복원
L3. 삭제 스냅샷실수 삭제삭제 시 일정 전체 데이터를 ourday_delete_log에 보존
L4. 동기화 보존동기화 충돌충돌 시 양쪽 데이터 모두 보존. 절대 덮어쓰기 금지
L5. 병합 검증데이터 소멸병합 후 일정 수가 줄면 쓰기 중단

13.2 save() 함수 프로세스

1. 현재 데이터 → backup에 복사→ 2. JSON.stringify→ 3. localStorage.setItem→ 4. 즉시 getItem으로 read-back→ 5. JSON.parse 후 비교→ 6. 불일치 시 backup에서 복원

13.3 삭제 프로세스

1. confirm 다이얼로그→ 2. 일정 전체 데이터를 deleteLog에 스냅샷 저장→ 3. events에서 제거→ 4. save() (백업+검증 포함)

15.3 일정 삭제 프로세스

[삭제] 클릭→ confirm 다이얼로그→ 취소 → 아무것도 안함
확인→ deleteLog에 스냅샷 저장→ events에서 제거→ 알림 생성→ save() (백업+검증)→ 모달 닫힘→ 달력 재렌더→ 토스트

15.4 .ics 익스포트 프로세스

설정 → [내보내기] 클릭→ events 전체 순회→ 각 일정 → VEVENT 변환→ 반복(RRULE) 매핑→ 알림(VALARM) 매핑→ VCALENDAR 래핑→ Blob + download

15.5 동기화 프로세스 (프로덕션)

앱 시작 / 포커스 복귀→ 로컬 백업 생성→ 서버 fetch→ 일정별 updatedAt 비교→ 병합 결과 생성→ 병합 검증 (수 감소 체크)→ 충돌 있으면 diff UI→ 로컬+서버 동시 쓰기→ 달력 재렌더
